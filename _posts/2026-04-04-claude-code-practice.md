> 我们这个月在几个模块里试了 Claude Code。不是评测，不是种草，只是记录一个管理者的观察：它到底省了什么时间，又在什么地方偷偷把成本加了回去。

## 一、最初的动机：我们想省掉什么

我的团队里，AI 代码贡献率已经不低了。但真正拖慢节奏的，往往不是"写代码"本身。

而是这三件事：

- **认知重建的损耗**。一个工程师一天要在"理解需求→换技术栈→切回老代码→Review别人"之间横跳好几次，每次都意味着重新加载上下文。
- **知识的板结**。项目规范、架构偏好、踩过的坑，分散在文档、注释和某些老员工的脑子里。新人进来或跨模块支援时，反复撞同一面墙。
- **环节的串行等待**。需求评审完等设计，设计完等排期，排期完等接口——信息在传递中层层失真。

Claude Code 吸引我的地方，不是它能替我写完一个功能，而是**它可能把"知识传递"和"环节并行"的成本打掉一截**。

## 二、实战中的四个发现

### 1. 提示词不是越全越好，而是越"硬"越好

早期我的系统提示词写得很长：技术栈、编码规范、Review标准、部署流程……恨不得塞成一本手册。结果是 Claude 常常选择性忽略后半段。

现在我把它压到 200 字以内，只放**不可触碰的红线**：
- 必须使用的框架和工具
- 分包和命名规则
- 测试覆盖率的最低要求

其他的，让它在需要时去查文档。提示词应该像护栏，不是百科全书。

### 2. Plan 模式真正有用的不是"拆分"，而是"对齐"

Claude Code 的 Plan 模式能把复杂需求拆成模块清单。这个功能本身并不难，任何一个有经验的 TL 都能做。它的真正价值在于：**拆分的过程可以被反复追问和修正，直到双方（人和 AI）的理解完全一致**。

这省掉的是后续返工的情绪成本。我见过太多 bug 不是因为写错了，而是因为开工前对"同义词"的理解不一样。

### 3. 子代理的"分工"很美，但边界很糊

我们实验了把任务拆成四个角色：架构师、实现者、Reviewer、前端配置器。理论上很美，实际上最大的坑是**上下文同步**。

- 架构师出的方案文档，实现者不一定完整加载
- Reviewer 提的问题，实现者可能只修了一半
- 错误会在角色之间传递并放大

目前可行的做法不是"全自动流水"，而是**把每个角色的输出当成一个需要人验收的中间产物**。人能快速扫一眼并叫停，这个环节目前省不掉。

### 4. SKILL 沉淀是唯一具有复利效应的事

Claude Code 的一个被低估的能力是 SKILL：把一次性的 Prompt 变成可反复调用的规则包。

我的团队已经在沉淀两类 SKILL：
- **工具接入标准**：比如 Elasticsearch 的查询规范、Redis 缓存更新策略
- **常见错误处理**：比如我们的分布式锁冲突应该怎么排查和修复

这些看上去不起眼，但解决了一个真实的问题：**把个人经验变成团队资产**。

## 三、我对"三阶段对话模型"的简化理解

网上流传的三阶段模型（需求定义→边界明确→迭代反馈）我试了，但实际运行时我只会抓三件事：

1. **说清楚用户故事和验收标准**。没有验收标准的需求，我不开工单给 Claude。
2. **明确区分"必须遵守"和"建议参考"**。后者经常会被 AI 当作可选项忽略，必须反复强调。
3. **在关键节点人工踩刹车**。表设计完成、核心业务逻辑完成、单元测试通过——这三个点必须停下来过一眼。

本质上，这不是"更好的提示词工程"，而是**管理者对节奏的控制**。

## 四、哪些地方我不让它碰

经过一个月的观察，我画了几条红线：

- **架构设计**：Claude 擅长组合已有的最佳实践，但突破性创新不是它的强项。涉及核心模块的改动，必须由人拍板。
- **跨模块依赖梳理**：它能识别显式依赖，但"牵一发而动全身"的隐性影响，往往需要人的全局视角。
- **线上质量决策**：灰度比例、回滚策略、监控阈值——AI 可以给出建议，但签字的是我。

还有一个越来越清晰的判断：**开发者对 AI 生成的代码负全部责任**。如果出了问题，背锅的不是模型，是最后一道 Review 它的人。这条底线必须让团队里的每个人都知道。

## 五、对团队的实际意义

### 解决"业务需求梳理不出来"的方向

"三阶段对话"在我们这里更大的价值不是提效，而是**逼业务方把模糊需求变成可验证的标准**。验收标准写不清楚的，AI 也写不出对的代码。这层倒逼意外的有用。

### 组织能力建设

子代理和 SKILL 本质上都是在做同一件事：**把不可见的个人经验变成可见的团队资产**。技术方案文档、错误案例库、Prompt 规则——这些东西以前只在某些人脑子里，现在至少在文件系统里有个落脚点。

### 衡量体系

下个月我会在团队里启动一个小实验：
- 记录 AI 生成代码的缺陷类型和频率
- 追踪"AI 辅助开发"实际节省的时间（不只是代码行数）
- 建立人机协作的效率基线

没有数据的判断都是叙事。我不靠叙事做管理。

## 六、写在最后的冷判断

Claude Code 没有改变一个基本事实：**最有价值的工程师不是写代码最快的人，而是最会把控质量、最能拆解复杂问题的人**。

AI 把时间分配的方式改了——写代码的占比在下降，设计和审查的占比在上升。这不是坏事，但团队的能力模型必须跟着调。

我的近期计划：
1. 把现有的 SKILL 库整理一遍，淘汰那些 AI 已经不再常犯的条目。
2. 在 APP 团队和 AI 工程化团队各选一个试点模块，跑完整的三阶段流程。
3. 一个月后用数据复盘，决定是扩大范围还是收紧边界。

还是那句话：**先想怎么不输，再想怎么赢。**
