读这篇文章的时候，我一直在点头。

不是因为作者说了什么新东西，恰恰相反——他把 AI Agent 工程中的那些混乱，翻译成了我管理团队时早就熟悉的问题。

这件事本身比任何新框架都重要。

---

## 确定性假设的崩塌

传统软件的可靠性链条——防错、观测、状态管理、验证、治理——全都建立在一个默认前提上：**系统是确定性的**。错误可以被穷举，行为可以被复现，产出可以被精确度量。

Agent 的概率性输出把这个前提砸碎了。

但作者没有停留在"这很难"的感叹上。他做了更重要的事：**把 Agent 的每个问题映射到组织管理中的对应场景**。

错误无法穷举？这就是跨部门协作中的组合性风险——你不会逐个审查每个接口，而是做端到端的流程审计。

产出无法度量？这就是考核指标失效——你不会死磕量化评分，而是引入定性评估。

Agent 感知到 context 压力后走捷径？这就是员工的 context anxiety——你不会等他们崩溃，而是提前合理分配工作量、明确优先级。

测试失效？这就是团队中产出波动大的成员——你不会靠单次考核定生死，而是建立持续的绩效追踪，用多次采样替代一次性判断。

这个映射让我松了口气。**原来我不是从零开始学一套全新的工程范式，只是把已经会的东西换了一个场景。**

---

## 但原则相通不等于直接搬

文章没有犯那个常见的错误："管理人和管理 Agent 是一样的"。

不一样。Agent Runtime 有自己的约束和机会，需要专门学习的新实践。

三个例子让我印象深刻：

**Document First 不是锦上添花，是基本前提。** 管理团队时，很多事情不用写下来，tribal knowledge 在空气里流动。Agent 没有这种隐性知识，所有上下文必须显式写成文档。这不是"最好有"，是"必须有"。

**Context Anxiety 需要主动管理生命周期。** 我的下属不会在任务做到 80% 的时候突然偷工减料——他们有余量意识，有职业操守。Agent 没有。当 context window 接近上限时，它会感知到压力并开始走捷径。你需要在阈值之前做 divide and conquer，而不是等它出错后再补救。

**Bootstrapping 在 Agent 场景下成本几乎为零。** 让十个人同时做同一件事然后赛马，成本高、伦理上也说不通。但十个 agent 并行跑同一个任务取最优结果，统计学上叫 bootstrapping，成本趋近于零，效果显著。这是 Agent 特有的机会，不是人的替代，是人的不可能变可能。

---

## 我的判断

这篇文章最打动我的不是它的技术深度，是它的视角选择。

它没有说"AI 是全新范式，你必须抛弃所有旧知识"——这是现在最常见的叙事，也是最让我疲惫的叙事。它说的是：**你已经具备了 harness engineering 的原则基础。**

管过人、管过项目、管过团队，这些经验不是被替代的对象，是被迁移的素材。

当然，新的工程实践要学。Agent runtime 的概率性输出、context window 限制、工具调用的不确定性、多 agent 状态同步——这些都是实实在在的新约束，需要专门的解法。

但原则层面的焦虑，可以放下了。

冷静的算术比热情的叙事更可靠。先想怎么不输——理解约束、承认边界、把已有的能力映射到新场景——再想怎么赢——学新的工程实践、抓 Agent 特有的机会。

这算是我最近读到最"解渴"的一篇文章。
