> 上个月团队在评估一个方案：让 AI Agent 自动生成前端界面。看了 Vercel 的 JSON Render 和 Google 的 A2UI，两条路线的差异比我想象中大得多。这篇文章记录我的选型判断。

## 它们想解决的问题是一样的

传统 AI 助手只能返回文本。当用户要填复杂表单、看数据图表、做多步骤操作时，纯文本对话就成了瓶颈。生成式 UI 的核心想法很简单：**让 AI 自己决定该展示什么界面，而不是开发者预先写死所有页面**。

但实现这个目标的哲学，Vercel 和 Google 走了完全相反的方向。

## Vercel JSON Render：完全自定义，但也完全自担

Vercel 的方案核心是三个概念：

- **Catalog（组件目录）**：开发者自己定义 AI 能用什么组件、每个组件有哪些 Props
- **Registry（注册表）**：把 Catalog 里的抽象定义映射成实际的 React 组件
- **Spec（规范）**：AI 生成 JSON 对象，客户端动态渲染

它的最大优点是**自由**。你的团队有什么自定义组件——复杂的图表、特殊的表单、品牌化的卡片——都可以放进去让 AI 调用。而且它和 Vercel AI SDK 深度集成，如果你已经在用 Next.js，上手成本很低。

但它的安全模型是"开发者自己负责"。通过 Zod Schema 做 Props 校验、用白名单限制事件名，这些措施有效，但前提是开发者配对了。配错一个 Props 类型，AI 就可能传进超长的字符串或危险的事件名。

还有一个隐形成本：**Catalog 的定义和维护**。组件越多，Catalog 越厚，AI 调用的精确度和可预测性就越难保证。

## Google A2UI：先锁死边界，再谈扩展

Google 的方案要保守得多。它只提供 18 个标准原语：Row、Column、Card、Text、Button、Table、Chart 等等。

这个设计的底层逻辑是：**AI 生成的不是可执行代码，而是一份纯声明式的 JSON 描述**。客户端收到后，把它映射成本地原生组件——React、Flutter、Angular、SwiftUI 都可以。因为描述本身不包含代码，XSS 和任意代码执行的风险从源头上就被消除了。

另外，A2UI 通常和 AG-UI（传输协议）配合。AG-UI 负责 SSE 流式传输、双向状态同步、工具调用事件。这套组合更像一个完整的企业级协议栈，而不是一个前端库。

代价也很明显：**18 个原语很难覆盖复杂的业务场景**。如果你的产品需要高度定制化的交互或品牌表达，A2UI 会显得很局促。

## 我的直接对比

| | Vercel JSON Render | Google A2UI |
|:---|:---|:---|
| 组件范围 | 完全自定义 | 18 个标准原语 |
| 灵活度 | 高 | 中 |
| 安全性 | 依赖 Schema 校验 | 声明式描述，无代码执行 |
| 生态绑定 | React/Next.js 为主 | Web/移动/桌面跨平台 |
| 上手成本 | 低（如果已经在 Vercel 生态） | 低（原语少，学习成本低） |
| 长期维护成本 | Catalog 越厚越难管 | 原语固定，但扩展受限 |

## 我的选型判断

**如果团队已经在 Next.js 生态，且 UI 高度定制**——比如智能家居 App 里那些带有实时设备状态动画的控制面板——Vercel JSON Render 更合适。但必须有一个人专门负责 Catalog 的质量和 Schema 的边界约束。

**如果产品对安全性要求极高，或者需要同时覆盖 Web、iOS、Android**——比如涉及健康数据展示或金融操作的界面——Google A2UI 是更稳妥的选择。你要接受的是，某些复杂交互可能需要拆成多步简单的原语组合。

**混合方案**其实更现实：标准化场景（表单、表格、图表）走 A2UI，品牌定制组件走 JSON Render。两者在同一应用里共存，不是非此即彼。

## 长期趋势的判断

生成式 UI 正在形成三层协议栈：
- UI 描述层（A2UI、JSON Render、未来可能的 MCP-UI）
- 传输协议层（AG-UI、A2A、WebSocket）
- 工具/上下文层（MCP、Function Calling）

目前各厂商都有自己的协议，统一标准短期内不会出现。这意味着**作为技术管理者，现阶段最重要的是把业务逻辑层和平台适配层解耦**，不管上层用哪个协议，底层能力都能快速迁移。

## 写在最后

Vercel 代表了一种"先给自由，再靠开发者自己约束"的 Web 原生哲学；Google 代表了一种"先把边界锁死，再考虑有限扩展"的企业级哲学。

我的判断是：**对于智能家居/IoT 这类涉及家庭安全和用户信任的领域，安全模型的优先级高于灵活度**。在这个前提下，A2UI 的保守不是缺点，而是合理的取舍。

当然，前提是它的 18 个原语能覆盖我们 80% 的场景。如果覆盖率低于这个数，再安全也没有意义。

先想怎么不输，再想怎么赢。
