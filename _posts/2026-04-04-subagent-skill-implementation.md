> 基于得物技术实践，结合团队现状设计的落地方案，适用于 AI 工程化团队能力建设。

## 现状分析

### 团队特点

| 维度 | 现状 | 影响 |
|------|------|------|
| 团队结构 | APP团队 + 系统测试团队 + AI工程化团队 | 跨团队协作是常态，需要标准化接口 |
| 已有AI场景 | 营销内容、食物录入、智能客服 | 点状落地，缺乏复用机制 |
| 基础设施 | CDP、行为埋点完备 | 数据基础好，可以支撑AI决策 |
| 核心痛点 | 业务需求梳理不清，责任错位 | 需要明确人机分工边界 |

### 与得物场景的异同

**相同点**：
- 都需要解决"需求理解偏差"问题
- 都有多个团队协作的需求
- 都需要把个人经验转化为组织资产

**不同点**：
- 得物是单一研发团队，跨团队（APP+测试+AI工程化）场景更复杂
- 得物聚焦后端开发，涉及移动端、AI模型、测试多个领域
- 已有AI场景需要整合而非从零建设

## 子代理系统设计

### 角色定义

```
┌─────────────────────────────────────────────────────────────┐
│                      协调者（人类）                           │
│                  项目经理/技术负责人                           │
└─────────────────────────────────────────────────────────────┘
                              ↓
        ┌─────────────────────┼─────────────────────┐
        ↓                     ↓                     ↓
┌───────────────┐    ┌───────────────┐    ┌───────────────┐
│  需求分析代理   │    │  技术方案代理   │    │  质量评估代理   │
│  (AI)          │    │  (AI)          │    │  (AI)          │
└───────────────┘    └───────────────┘    └───────────────┘
        ↓                     ↓                     ↓
┌───────────────┐    ┌───────────────┐    ┌───────────────┐
│ • 需求拆解     │    │ • 技术选型     │    │ • 测试策略     │
│ • 验收标准     │    │ • 接口设计     │    │ • 风险评估     │
│ • 场景梳理     │    │ • 模块划分     │    │ • 合规检查     │
└───────────────┘    └───────────────┘    └───────────────┘
                              ↓
        ┌─────────────────────┼─────────────────────┐
        ↓                     ↓                     ↓
┌───────────────┐    ┌───────────────┐    ┌───────────────┐
│  APP实现代理   │    │  AI模型代理    │    │  测试执行代理   │
│  (AI)          │    │  (AI)          │    │  (AI)          │
└───────────────┘    └───────────────┘    └───────────────┘
```

### 协作流程

**阶段一：需求澄清（1-2天）**
业务方提出需求 → 协调者分配给需求分析代理 → 输出结构化需求文档 → 组织评审 → 确认需求

**阶段二：技术方案（2-3天）**
协调者分配给技术方案代理 → 输出技术方案文档 → 质量评估代理风险评估 → 技术评审 → 确认方案

**阶段三：并行实现**
协调者创建任务 → AI模型代理负责模型训练和部署 → APP实现代理负责客户端集成 → 测试执行代理负责测试用例编写

**阶段四：集成验收（2-3天）**
各模块集成测试 → 质量评估代理验收测试 → 验收评审 → 上线或回退

## SKILL机制设计

### 什么是SKILL

SKILL是将**有效的Prompt指令沉淀为可复用的标准化资产**。类比：
- 普通开发：每次遇到ES查询都重新写代码
- 有SKILL：直接调用"ES查询最佳实践"SKILL，AI按标准方式实现

### 需要的SKILL清单

#### 通用层SKILL（跨场景复用）

| SKILL名称 | 适用场景 | 内容概要 |
|-----------|----------|----------|
| `api-design` | 所有API设计 | 接口命名规范、参数校验规则、错误码定义 |
| `data-pipeline` | 数据相关 | CDP数据接入规范、埋点设计规范 |
| `model-deploy` | AI模型部署 | Triton Server配置、模型版本管理、A/B测试 |
| `mobile-integration` | APP集成 | 移动端API调用规范、缓存策略、错误处理 |

#### 场景层SKILL（特定场景复用）

| SKILL名称 | 适用场景 | 内容概要 |
|-----------|----------|----------|
| `content-generation` | 营销内容生成 | 内容模板、风格控制、审核规则 |
| `food-recognition` | 食物录入 | 图像预处理、识别后处理、数据校准 |
| `intent-classification` | 智能客服 | 意图分类模型微调、多轮对话管理 |

#### 工具层SKILL（开发效率提升）

| SKILL名称 | 适用场景 | 内容概要 |
|-----------|----------|----------|
| `code-review` | 代码审查 | 代码规范检查、常见错误识别、优化建议 |
| `test-case-gen` | 测试用例生成 | 根据需求生成测试用例、边界条件识别 |
| `doc-gen` | 文档生成 | API文档、技术方案文档、操作手册 |

### SKILL标准格式

```yaml
# SKILL: api-design
# 版本: 1.0
# 更新日期: 2026-03-04

## 1. 适用场景
设计新的API接口时使用

## 2. 输入要求
- API的功能描述
- 请求/响应的数据结构
- 特殊约束

## 3. 输出规范
必须包含：
- 接口路径（RESTful风格）
- 请求参数（含校验规则）
- 响应结构（含错误码）
- 接口文档（Markdown格式）

## 4. 约束条件
- 接口路径使用 /api/v{版本}/{模块}/{动作} 格式
- 统一返回格式：{ "code": 0, "data": {}, "message": "ok" }

## 5. 示例
[完整示例]

## 6. 常见错误
- ❌ 错误：返回格式不统一
- ✅ 正确：统一使用 ResultDTO 包装

## 7. 变更历史
- 2026-03-04: 初始版本
```

## 实施路线图

### 第一阶段：试点（1-2个月）

**目标**：验证子代理系统和SKILL机制的有效性

**选择试点项目**：智能客服意图识别优化或营销内容生成

**建设核心SKILL**（3-5个）：
1. `api-design` — 最通用，所有项目都用得上
2. `model-deploy` — AI工程化团队核心能力
3. 1-2个场景层SKILL（根据试点项目）

**建立协作流程**：
- 跑通"需求澄清→技术方案→并行实现→集成验收"四阶段
- 记录每个阶段的耗时和问题
- 形成第一版《子代理协作操作手册》

### 第二阶段：推广（3-4个月）

- 覆盖所有AI工程化团队的项目
- 逐步引入APP团队和测试团队
- 根据反馈优化现有SKILL
- 建立衡量体系：需求理解准确率、开发效率提升、缺陷率变化

### 第三阶段：制度化（5-6个月）

- 更新团队工作流程文档
- 新成员入职培训包含子代理系统和SKILL使用
- 定期（每季度）回顾和优化

## 风险与应对

| 风险 | 可能性 | 影响 | 应对措施 |
|------|--------|------|----------|
| 团队抵触新流程 | 中 | 高 | 从志愿者开始试点，用数据证明价值 |
| SKILL维护成本高 | 中 | 中 | 建立SKILL负责人制度，定期Review |
| AI输出质量不稳定 | 高 | 中 | 保留人工审核环节，渐进式信任 |
| 跨团队协作不畅 | 中 | 高 | 明确接口和交付物标准，定期同步 |
